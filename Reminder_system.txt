SO ACCORDING TO THE QUESTION , WE HAD TO IMPLEMENT A REMINDER SYSTEM BASED ON TWO CONDITIONS
### CONDITION 1 --->IF STATUS IS COMPLETE , THEN NO NEED OF IMPLEMENTING REMINDER SYSTEM
### CONDITION 2----> IF STATUS IS PENDING , THEN FIRST CHECK THE DUE_DATE , IF DUE_DATE IS LESS THAN DATE.NOW() THEN IMPLEMENT REMINDER SYSTEM ELSE NO NEED


### ARCHITECTURE OF MY REMINDER SYSTEM
SO NORMALLY TO BUILD A REMINDER WE CAN USE OTHER LIBRARY LIKE BULLMQ BUT I AM NOT AWARE OF IT , ALSO I WANT TO SHOW MY LOGICAL THINKING 
### DUE TO ALL THIS I BUILT A DIFFERENT SIMPLE SMALL SCALE UNIQUE REMINDER SYSTEM USING MAP() AND DATABASE

### MY LOGIC
WHAT I NOTICED THAT I HAVE TO OR SERVER HAVE TO , DEAL WITH NOTICE ONLY FOR THREE APIS
### API 1---->POST api/tasks--> DURING CREATION
SO WHAT I DID IS THAT WHEN USER WILL SEND ITS TASK FORM , I WILL QUICKLY CHECK WHETHER ITS STATUS IS PENDING OR COMPLETE ,IF COMPLETE NP, BUT IF PENDING THEN
I WILL CHECK DUE_DATE , IF DUE_DATE<NOW() I WILL IMPLEMENT A SETTIMEOUT( (CALLBACK)=>{ CONSOLE.LOG("REMINDER FOR YOUR TASK"};CALLBACK();},DUE_DATE-NOW()-1);
AND STORE IT IN A MAP() USING TASKID AS KEY AND SETTIMEOUT OBJECT AS VALUE , BUT HERE COMES THE LOGIC PART , WE KNOW THIS WILL STAYS IN MEMORY ONLY UNTIL MY SERVER
IS RUNNING , SO WHAT I DID I CREATED A TASK_REMINDERS TABLE , WHICH WILL STORE AL SUCH TASK_ID WHICH HAVE GOT REMINDERS IN IT WITH THEIR DUE_DATE
NOW WHEN THE APP OR SERVER WILL CRASH , AFTER RESTARTING , I WILL FIRST FETCH ALL THAT TASK AND MATCH THAT WHETHER THEIR DUE_DATE IS STILL VALID WHICH MEANS < NOW()
IF YES I WILL SET THEIR TIMEOUT AGAINS IN THE MEMORY 


### API 2----> PUT api/tasks/:id ---> during updating
DURING UPDATING WHEN SERVER WILL RECIEVES THE FORM , IT WILL FIRST FETCH ALL DATA RELATED TO THAT TASK_ID FROM DATABASE AND THEN IT WILL MATCH STATUS AND DUE_DATE
IF STATUS IS COMPLETE , IT WILL SIMPLY CALL DELETE QUERY TO TASK_REMINDERS TABLE, IF THERE WAS TASK_ID IT WOULD BE DELETED ELSE NOTHING WILL HAOPPEN , FOR DUE_DATE , IF
DUE_DATE IS CHANGED WE WILL UPDATE IN REMINDER TABLE AND ALSO DELETE IT FIRST FROM MAP() AND THEN ADD NEW ONE


### API 3---> DELETE api/tasks/:id
THIS WILL SIMPLY CALL DELETE QUERY ON REMINDER TABLE , IF ENTRY WAS THERE IT WOULD BE DELETED ELSE NOTHING WILL HAPPEN
SAME FOR MAP() IT WILL DELETE TASK_ID FROM MAP
